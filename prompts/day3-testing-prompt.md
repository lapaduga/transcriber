# Day 3 Testing — Промпт для агента

## Контекст проекта

**Транскриптор** — Flask-приложение для транскрипции видео/аудио через Whisper + DeepSeek AI.
Два сервера: `server.py` (API, порт 3000) + `mcp_server.py` (ML-воркер, порт 3001).
Фронтенд: vanilla JS (app.js, voice.js, tts.js), без фреймворков.
Состояние: in-memory (conversations, stream_results, jobs), нет БД.

## Миссия

Подготовить два уровня тестирования для проекта Транскриптор.

## УРОВЕНЬ 1: Unit/Integration тесты на код

### Задача агенту

1. **Проанализировать кодовую базу** и найти непокрытые модули
2. **Написать тесты** (pytest) для минимум 3 модулей
3. **Запустить тесты** и убедиться что проходят с первого запуска

### Какие модули тестировать (приоритет)

#### Модуль 1: `server.py` — утилиты и валидация
- `sanitize_filename()` — замена опасных символов, обрезка, дефолт
- `allowed_file()` — проверка расширений
- `_sanitize_for_tts()` — очистка Markdown для TTS
- `_format_duration()` — форматирование секунд в "Xh Ym Zs"

#### Модуль 2: `server.py` — Flask routes (интеграционные)
- `GET /api/health` — проверка статуса MCP/Vosk/DeepSeek
- `GET /api/stats` — CPU/RAM/GPU метрики
- `POST /api/upload` — валидация файла, размер, расширение
- `POST /api/chat` — создание чата, tool calling
- `POST /api/summarize` — саммари через DeepSeek
- `POST /api/cancel/<id>` — отмена задачи

#### Модуль 3: `mcp_server.py` — MCP worker
- `GET /tools` — список доступных инструментов
- `POST /call` — создание задачи транскрипции
- `GET /job/<id>` — получение статуса задачи
- `POST /cancel/<id>` — отмена задачи

#### Модуль 4 (бонус): JS утилиты через Node.js
- `parseMarkdown()` — парсинг Markdown в HTML
- `formatElapsed()` — форматирование времени
- `formatDuration()` — форматирование длительности

### Стратегия мокирования

| Зависимость | Где используется | Стратегия мока |
|-------------|-----------------|----------------|
| `httpx` (DeepSeek API) | `server.py:_deepseek()` | `unittest.mock.patch` на `httpx.post` |
| `faster_whisper.WhisperModel` | `mcp_server.py:get_model()` | Мокать `model.transcribe()` |
| `edge_tts.Communicate` | `server.py:tts()` | Мокать `stream_sync()` |
| `psutil` | `server.py:stats()` | Мокать `cpu_percent`, `virtual_memory` |
| `subprocess` (nvidia-smi) | `server.py:stats()` | Мокать `check_output` |
| MCP subprocess | `server.py:start_mcp_server()` | Мокать `Popen` |
| Файловая система (uploads/) | Несколько файлов | `tmp_path` fixture |

### Требования к тестам

- Фреймворк: **pytest**
- Конфигурация: `[tool.pytest.ini_options]` в `pyproject.toml`
- Фикстуры: `conftest.py` с общими моками и test client
- Все тесты должны проходить **с первого запуска**
- Минимум **3 файла тестов**: `test_server_utils.py`, `test_server_routes.py`, `test_mcp_server.py`
- Покрытие: минимум 80% покрытие тестируемых функций

## УРОВЕНЬ 2: Smoke-сценарии через Playwright

### Описание сценариев (текстом)

#### Сценарий 1: Загрузка страницы и проверка приветствия
1. Открыть `http://localhost:3000`
2. Проверить что заголовок "Транскриптор" отображается
3. Проверить что welcome-иконка и описание видны
4. Проверить что статус-индикатор показывает "Проверка..." или "Все системы готовы"
5. **Скриншот**: главная страница

#### Сценарий 2: Создание нового чата и отправка сообщения
1. Нажать кнопку "+" (новый чат) в сайдбаре
2. Проверить что чат появился в списке слева
3. Ввести текст "Привет, это тест" в поле ввода
4. Нажать Enter или кнопку отправки
5. Проверить что сообщение пользователя отображается в чате
6. Проверить что появляется индикатор "думает" (thinking bubble)
7. **Скриншот**: чат с отправленным сообщением

#### Сценарий 3: Удаление чата
1. Создать новый чат
2. Отправить в него сообщение
3. Найти кнопку удаления (X) на чате в сайдбаре
4. Нажать удалить
5. Проверить что чат исчез из списка
6. **Скриншот**: сайдбар после удаления

#### Сценарий 4: Загрузка файла через drag-and-drop
1. Создать новый чат
2. Перетащить файл в зону drop (или кликнуть кнопку добавления файла)
3. Проверить что превью файла отображается (имя файла, кнопка удаления)
4. Проверить что placeholder изменился на "Нажмите Enter..."
5. Удалить файл через кнопку X
6. Проверить что превью скрылось
7. **Скриншот**: превью загруженного файла

#### Сценарий 5: Панель статистики системы
1. Открыть приложение
2. Проверить что правая панель "Система" видна
3. Проверить что CPU и RAM значения обновляются (не "--")
4. **Скриншот**: панель статистики с данными

### Требования к Playwright тестам

- Фреймворк: **Playwright** (Python, `pytest-playwright`)
- Браузер: **Chromium** (headless)
- Скриншоты: сохраняются в `test-reports/screenshots/`
- Все сценарии автоматизированы
- Тесты не требуют бэкенд для ML (мокаем API responses)

## ОГРАНИЧЕНИЯ (от экспертов)

### Аналитик
- Тестировать бизнес-логику, а не implementation details
- Каждый тест = один сценарий (single responsibility)
- Имена тестов: `test_<action>_<condition>_<expected>`

### Специалист по LLM/MCP
- Мокать DeepSeek API — никогда не вызывать реальный API в тестах
- MCP server тестировать отдельно от main server
- Tool calling тестировать через mock ответов DeepSeek

### Специалист по железу
- Тесты не должны требовать GPU
- Тесты не должны загружать Whisper модель
- Время выполнения全套 < 30 секунд

### Специалист по безопасности
- Нет секретов в тестах (API ключи через env/mocks)
- Тесты валидации файлов: попытки path traversal, oversized files
- Проверка что `_delete_file` не удаляет файлы за пределами UPLOAD_DIR

### Фронтендер
- JS тесты через Node.js (без сборщика)
- Парсинг Markdown — критичная функция, полное покрытие
- Нет依赖 на DOM в unit-тестах (чистые функции)

### Бекендер
- Flask test client для интеграционных тестов
- Каждый route: happy path + error case
- Thread safety: тесты с concurrent access

### UX/UI дизайнер
- Smoke-тесты покрывают основные user journeys
- Скриншоты на каждом шаге
- Визуальная проверка: элементы на месте, стили корректны
